=========================================================
This directive tells Claude Code exactly what to change in the existing JobFlow OS codebase.
Do NOT rebuild anything that already works. Only add or modify what is listed here.
Read every section before touching a single file.

PROBLEM BEING SOLVED
--------------------
Currently every tool call inside an agent rebuilds context from scratch —
loads profile, loads role, loads prefs — then discards it all when done.
By the time the message agent drafts, it has lost why the person was chosen,
what was noticed on their profile, what the hook should be.
Result: generic messages.

Fix: one Python dict (the context object) built ONCE at agent startup,
passed through every tool call, and grown at each step.
Never rebuilt mid-agent. Checkpointed to SQLite after every tool call
so the agent can resume if it crashes.

CHANGE 1 — CREATE NEW FILE
═══════════════════════════════════════════════
File to create: backend/memory/context_object.py

Tell Claude Code:
  Create this file. It defines the context object structure and all
  functions that build, update, checkpoint, and restore it.

  SECTION A — build_initial_context(job_id, session_id) -> dict
    Builds the context object ONCE at agent startup.
    Calls existing functions from database.py and decision_log.py.
    Returns a dict with exactly these 5 top-level keys:

    "permanent" — dict, never mutated after build:
      profile:        full text of profile/master_profile.md
      tone_guide:     full text of profile/tone_guide.md
      role:           result of get_job(job_id)
      enrichment:     result of get_enrichment_by_job(job_id)
      story:          text of stories/{company}_{role}.md if exists, else ""
      preferences:    result of get_preference_summary()
      prior_contacts: result of get_contacts_by_job(job_id)
                      stored as set of linkedin_urls for O(1) duplicate check

    "session" — dict, updated during run:
      session_id:            int
      agent_id:              str, set when agent starts
      standing_instructions: str, default ""
      candidates_found:      list, grows as search runs
      actions_taken:         list of {step, person, timestamp}

    "current_person" — dict, RESET between profiles, NOT between tool calls:
      name, title, linkedin_url, headline, summary,
      past_roles (list), mutual_connections (int),
      connection_degree, recent_activity
      All default to None or [] or 0.

    "current_score" — dict, RESET between profiles:
      score (int or None), reason (str), fit_angle (str), hook_source (str)
      All default to None.

    "observations" — list, RESET between profiles:
      Each item is a dict: {step (str), category (str), note (str)}
      Categories are fixed (see CHANGE 2).
      If nothing fits a fixed category, use category="freeform".
      Default: empty list.

    If profile/master_profile.md does not exist:
      raise FileNotFoundError with message:
      "master_profile.md not found. Create profile/master_profile.md before running agents."

    If enrichment does not exist for this job_id:
      raise ValueError with message:
      "No enrichment found for job_id {job_id}. Run enrichment agent first."

  SECTION B — reset_for_next_person(ctx) -> dict
    Resets ONLY current_person, current_score, and observations to defaults.
    Does NOT touch permanent or session.
    Call this between each LinkedIn profile — not between tool calls.
    Returns the modified ctx.

  SECTION C — checkpoint(ctx) -> None
    Writes ctx to SQLite agent_checkpoints table (see CHANGE 3).
    Serialises ctx to JSON with json.dumps().
    Called after every tool call that modifies ctx.
    If SQLite write fails: log warning, do NOT raise — never let a checkpoint
    failure crash the agent.

  SECTION D — restore_checkpoint(agent_id, job_id, session_id) -> dict or None
    Reads the latest checkpoint for this agent+job+session from SQLite.
    Deserialises with json.loads().
    Returns the dict if found, None if not found.
    Called at agent startup BEFORE build_initial_context.
    If a checkpoint exists: return it and skip build_initial_context.
    If no checkpoint: proceed with build_initial_context normally.

  SECTION E — is_duplicate(linkedin_url, ctx) -> bool
    Checks if linkedin_url is already in ctx["permanent"]["prior_contacts"].
    prior_contacts is stored as a Python set of URLs for O(1) lookup.
    Also checks ctx["session"]["candidates_found"] for within-session dupes.
    Returns True if duplicate, False if new.
    This is the ONLY place duplicate checking happens. Do not check elsewhere.

  SECTION F — mark_contacted(linkedin_url, ctx) -> dict
    Adds linkedin_url to ctx["permanent"]["prior_contacts"] set.
    Also appends to ctx["session"]["actions_taken"].
    Call this immediately after a successful send — not after approval.
    Returns modified ctx.

DUPLICATE PREVENTION — HOW IT WORKS END TO END
═══════════════════════════════════════════════
For Claude Code to understand the full picture:

Check 1 — on profile read (within a session):
  is_duplicate() checks ctx["permanent"]["prior_contacts"] set.
  prior_contacts is loaded from the contacts SQLite table at context build time.
  If a linkedin_url already exists in contacts table: skip silently.
  Never surface a person who was already contacted in any prior session.

Check 2 — within the same session's candidate list:
  is_duplicate() also checks ctx["session"]["candidates_found"].
  If the same profile appears twice in search results: skip the second instance.

Check 3 — after send:
  mark_contacted() adds the URL to ctx["permanent"]["prior_contacts"] immediately.
  So if two agents running in parallel both find the same person,
  the second agent's is_duplicate() check will catch it mid-session.
  This is why mark_contacted() must be called instantly after send —
  not at session end, not at the next checkpoint — IMMEDIATELY.

Check 4 — parallel agent safety:
  Two agents running simultaneously both write to the same contacts table.
  Before any agent calls insert_contact(), it must:
    SELECT COUNT(*) FROM contacts WHERE linkedin_url = ? AND job_id = ?
  If count > 0: skip. Do not insert. Do not send.
  This is the database-level safety net that catches race conditions
  that the in-memory set cannot catch between parallel threads.
  Add this check to insert_contact() in database.py as a guard:
    Wrap the INSERT in:
      IF NOT EXISTS (SELECT 1 FROM contacts WHERE linkedin_url=? AND job_id=?)
